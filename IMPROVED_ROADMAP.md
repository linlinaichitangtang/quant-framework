# OpenClaw 量化交易框架 — 改进 Roadmap

> 基于用户视角分析的改进规划
> 编写日期：2026-05-16

## 背景

本文档基于四个用户视角的评估结果编写：
1. **量化交易小白** — 初次接触量化交易
2. **用过量化的人** — 有一定量化经验
3. **老油条** — 专业量化交易者
4. **研究员** — 量化研究领域

---

## 改进优先级总览

| 优先级 | 阶段 | 主题 | 目标用户 |
|--------|------|------|----------|
| P0 | 快速修复 | 新手友好 | 小白 |
| P1 | 体验提升 | 实用功能 | 用过量化的人 |
| P2 | 专业扩展 | 高级功能 | 老油条 / 研究员 |
| P3 | 长期演进 | 前沿探索 | 研究员 |

---

## P0 — 新手友好 ⬜

### 目标：让小白 5 分钟内跑通 Demo

#### R0.1 — 一键启动脚本

**问题**：小白不懂 Docker 命令，不知道如何启动

**交付物**：
- [ ] `start.sh` 一键启动脚本（检测环境 + docker-compose up + 健康检查）
- [ ] `stop.sh` 一键停止脚本
- [ ] `reset.sh` 重置环境脚本
- [ ] 启动成功后的 Web UI 地址提示

**验收标准**：`./start.sh` 后，5 分钟内浏览器能访问 Demo

---

#### R0.2 — 新手教程文档

**问题**：小白不知道策略怎么配置、怎么运行

**交付物**：
- [ ] `QUICKSTART.md` — 5 分钟入门指南
- [ ] `TUTORIAL.md` — 完整使用教程（选股 → 运行 → 查看结果）
- [ ] 视频教程链接（YouTube/Bilibili）
- [ ] FAQ 文档（常见问题解答）

**验收标准**：小白能独立完成第一次选股和回测

---

#### R0.3 — 策略命名去色情化

**问题**："约炮式"、"恋爱式"策略命名不专业，小白误解

**交付物**：
- [ ] 重命名 `a_stock_evening.py` → `a_stock_short_term.py`
- [ ] 重命名 `us_hk_event_driven.py` → `us_hk_options_event.py`
- [ ] 更新所有文档和注释中的命名
- [ ] API 和前端同步更新

**验收标准**：策略名称中性、专业、易懂

---

#### R0.4 — 模拟盘演示模式

**问题**：小白想试又怕亏钱，不敢动手

**交付物**：
- [ ] Demo Mode 开关（界面一键开启）
- [ ] 演示模式账号（预设假资金 100 万）
- [ ] 模拟交易记录（不真实下单）
- [ ] 演示数据引导（引导小白体验完整流程）

**验收标准**：小白能在演示模式下完整体验选股 → 信号 → 回测

---

## P1 — 体验提升 ⬜

### 目标：提升已用过量化交易用户的体验

#### R1.1 — 统一 API 文档

**问题**：功能多但文档分散，小白找不到

**交付物**：
- [x] Swagger UI 访问说明（`/docs` 端点）
- [x] API 文档首页增强（展示所有端点和用法）
- [x] Postman 导入文件（`docs/OpenClaw-API.postman_collection.json`）
- [x] Python SDK 使用示例

**验收标准**：用户打开 `/docs` 就能知道每个 API 怎么用

---

#### R1.2 — 信号追溯功能

**问题**：老油条想知道每条信号背后的选股理由

**交付物**：
- [x] `SignalReasonResponse` Schema 定义
- [x] 信号详情 API（`GET /signals/{id}/reason`）
- [x] 前端信号详情弹窗（展示选股理由）- 待前端配合
- [x] 回测信号追溯（每笔交易关联选股理由）

**验收标准**：每条信号都有完整的决策依据说明

---

#### R1.3 — 回测详细报告导出

**问题**：老油条需要导出回测数据做进一步分析

**交付物**：
- [x] CSV 导出（交易明细）
- [x] Excel 导出（含图表）
- [x] PDF 导出（格式化报告）
- [x] JSON 导出（程序化读取）

**验收标准**：回测结果能导出为四种常用格式

---

#### R1.4 — 实盘/模拟盘一键切换

**问题**：用户想先模拟盘测试，再切实盘，操作麻烦

**交付物**：
- [ ] 交易环境开关（界面一键切换）
- [ ] 环境隔离（模拟/实盘账户完全独立）
- [ ] 环境状态指示器（当前所处环境）
- [ ] 切换环境确认（防止误操作）

**验收标准**：用户 3 秒内完成模拟盘/实盘切换

---

#### R1.5 — 多渠道告警

**问题**：现有告警只有 WebSocket，老油条希望能发钉钉/飞书/Telegram

**交付物**：
- [x] 钉钉 Webhook 通知（已有）
- [x] 飞书 Webhook 通知
- [x] Telegram Bot 通知
- [ ] 告警模板自定义
- [ ] 告警频率控制（避免轰炸）

**验收标准**：用户能配置任意一个渠道接收告警

---

## P2 — 专业扩展 ⬜

### 目标：满足专业交易者和量化研究需求

#### R2.1 — 自定义策略 DSL

**问题**：老油条想用自定义策略，但改代码太麻烦

**交付物**：
- [x] 策略描述语言（YAML/JSON 配置文件）
- [x] 条件表达式引擎（支持均线、涨幅、量比等条件）
- [x] 策略模板市场（用户分享模板）
- [x] DSL 文档和示例

**验收标准**：用户无需写代码，通过配置实现自定义选股策略

---

#### R2.2 — 实时 WebSocket 替代轮询

**问题**：定时轮询数据延迟高，行情波动时跟不上

**交付物**：
- [x] 富途 WebSocket 行情接入（模拟数据 + 接口框架）
- [x] WebSocket 实时 K 线推送
- [x] WebSocket 实时成交推送
- [x] 前端实时行情组件（接口已暴露）

**验收标准**：行情数据延迟 < 1 秒

---

#### R2.3 — 策略参数热更新

**问题**：改参数要重启服务，影响实盘

**交付物**：
- [x] 运行时参数修改 API
- [x] 参数变更历史记录
- [x] 参数验证（修改前检查合法性）
- [x] 原子更新（避免状态不一致）

**验收标准**：修改参数不影响正在运行的策略

---

#### R2.4 — 更多券商支持

**问题**：只支持 FMZ，老油条想用 IB/Optiver

**交付物**：
- [x] Interactive Brokers (IB) API 适配器框架
- [x] 统一执行层接口（适配器模式）
- [x] 多券商账户管理
- [x] 券商切换 API

**验收标准**：一套策略能在不同券商执行

---

#### R2.5 — 阿尔法因子库集成

**问题**：只有 51 个因子，研究员需要更多

**交付物**：
- [x] 阿尔法因子库基础框架
- [x] 常用因子定义（均线、动量、量比、波动率等）
- [x] 因子 IC/IR 分析
- [x] 因子组合回测
- [x] 自定义因子注册接口

**验收标准**：研究员能快速分析新因子的有效性

---

#### R2.6 — 多空策略模板

**问题**：只有做多策略，缺少对冲手段

**交付物**：
- [ ] Long/Short Equity 策略模板
- [ ] Market Neutral 策略模板
- [ ] 配对交易策略模板
- [ ] 融券做空支持（券商接口）

**验收标准**：用户能配置一个多空对冲组合

---

#### R2.7 — 统计套利模块

**问题**：缺少协整性检验、均值回归等高级策略

**交付物**：
- [ ] 协整性检验（Engle-Granger / Johansen）
- [ ] 均值回归策略模板
- [ ] 价差均值回归策略
- [ ] 跨品种套利检测

**验收标准**：系统能自动检测套利机会

---

#### R2.8 — 时序交叉验证框架

**问题**：回测容易过拟合，研究员需要防数据泄露

**交付物**：
- [x] 时间序列拆分器（TimeSeriesSplit）
- [x] Walk-Forward 分析
- [x] 未来函数检测
- [x] 样本外测试报告

**验收标准**：回测报告包含样本外测试结果

## 版本规划

| 版本 | 阶段 | 交付内容 | 预计周期 |
|------|------|----------|----------|
| R0.1 ~ R0.4 | P0 新手友好 | 一键脚本 + 教程 + 演示模式 | 2 周 |
| R1.1 ~ R1.5 | P1 体验提升 | API文档 + 信号追溯 + 导出 + 切换 + 告警 | 3 周 |
| R2.1 ~ R2.8 | P2 专业扩展 | DSL + WebSocket + 热更新 + 多券商 + 因子库 | 4 周 |
| R3.1 ~ R3.5 | P3 前沿探索 | 贝叶斯 + RL环境 + 高频 + 云调度 | 4 周 |

---

## 开发规范

1. **优先级排序**：先 P0 再 P1，P2/P3 按需开发
2. **用户测试**：每个 P0 功能必须由小白用户测试通过
3. **向后兼容**：所有改进不破坏现有 API
4. **文档先行**：功能开发前先写文档
5. **测试覆盖**：新功能单元测试覆盖率 >= 85%
